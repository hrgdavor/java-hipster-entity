## Context

The project needs a lightweight runtime bridge that materializes view interfaces from entity-backed data without requiring generated concrete classes for every view. This should support quick adoption while preserving the interface-first contract.

## Decision

Use dynamic proxy strategy for read-oriented view interfaces with metadata-driven property resolution:

**Core runtime pieces:**
- `ViewProxyFactory`: creates runtime proxy instances for a target view interface
- `ViewInvocationHandler`: maps method calls to field metadata entries
- `EntityValueReader`: abstraction responsible for reading values from underlying entity source
- `PropertyResolver`: resolves view method to canonical property id

**Method mapping rules:**
- Record-style accessors are canonical (`id()`, `firstName()`), not JavaBean getters
- `Object` methods are handled explicitly (`toString`, `hashCode`, `equals`)
- Strict mode by default: unknown property method throws descriptive exception rather than silently returning null

**Source abstractions:**
- Entity object with generated metadata
- Array-backed row (`Object[]` + ordinal mapping)
- Unified reader contract: read by property id with optional typed helper and optional presence check for partial projections

## Consequences

- Avoids generating many classes for early-stage usage, minimizing source churn.
- Supports broad flexibility while contracts settle.
- Enables one-line materialization from source to view: `viewProxyFactory.create(PersonSummary.class, entitySource)`.
- Expected costs: reflective or method-handle dispatch overhead per accessor, potential boxing/casting overhead, less JIT optimization versus concrete classes.
- Mitigations available: cache method to property mapping at proxy construction; prefer `MethodHandle` over raw reflection; expose path to generated concrete adapter for hotspots.

## Out of scope

- Write/update mutation semantics — covered by DEC-012.
- Optional factory-module policy for proxy vs generated replacement — covered by DEC-013.
- Compile-time metadata generation details — already covered by DEC-004 and DEC-009.

## Acceptance criteria

- `ViewProxyFactory` MUST support creating proxy instances for arbitrary view interfaces.
- Property method dispatch MUST be deterministic and include clear diagnostics for missing/unknown methods.
- Source abstraction MUST support both entity-backed and array-backed readers.
- Debuggable `toString` MUST expose mapped property set.
- Strict-mode behavior MUST prevent silent data corruption from unknown methods.
