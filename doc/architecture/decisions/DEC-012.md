## Context

Builder workflows need efficient mutable storage during composition and update flows. The project needs clear semantic models for overwrite, presence tracking, dirty flag behavior, and patch-oriented persistence.

## Decision

Provide two array-backed storage primitives with explicit change-tracking semantics:

**EntityUpdateArray (baseline):**
- Store latest assigned values by property ordinal
- Overwrite semantics: last write wins
- No intrinsic knowledge of whether value changed from original
- Suitable for full replace workflows

**EntityUpdateTrackingArray (tracking variant):**
- Track whether fields were touched during builder operations
- Enable patch-oriented writes and selective persistence
- Two specialized implementations based on field count:
  - `EntityUpdateTrackingArray64`: single `long` field for entities with fewer than 64 fields (no array allocation)
  - `EntityUpdateTrackingArray`: `long[]` for entities with 64 or more fields

**Dirty tracking behavior:**
- Touched flag tracked via bitmask indexed by field ordinal
- Explicit null assignment: touched bit set with null value
- Same-value assignment is a no-op: does not set the touched/updated bit
- Changed-value assignment sets the touched/updated bit

**Merge behavior:**
Explicit mode naming required to avoid ambiguity:
- Overwrite-all: incoming values always replace
- Non-null overwrite: only non-null incoming values replace
- Touched-only merge: only fields marked touched in source apply

## Accepted notes

All four storage classes implemented in `hipster-entity-core`:
- `EntityUpdateArray` — baseline overwrite semantics
- `EntityUpdateTrackingArray` — abstract base with bitmask change tracking
- `EntityUpdateTrackingArray64` — single `long` bitmask for ≤64 fields (concrete subclass)
- `EntityUpdateTrackingArrayLarge` — `long[]` bitmask for >64 fields (concrete subclass)

Factory method `EntityUpdateTrackingArray.create(enumClass, values)` selects concrete variant automatically based on field count. Proxy handler (`UpdatableHandler` in `ArrayBackedViewProxyFactory`) routes `changes()` and `clearChanges()` to the tracking array; other zero-arg calls go through `FieldNameMapper`.

## Consequences

- Provides efficient mutable storage without redundant boolean arrays (bitmask as canonical touched tracking).
- Supports both full-replace and patch-update workflows with the same interface.
- Compact single-word representation for common case (64 fields or fewer).
- Explicit merge modes prevent ambiguity in update semantics.
- Tight coupling between array variant and field count; transition between variants requires recreation.
- Adds memory and logic overhead versus baseline (EntityUpdateArray) due to change tracking.

## Out of scope

- Converter implementation or transformation pipeline (handled separately).
- Write-path SQL generation or query planning (integration layer concern).
- Diagnostics and validation beyond change-tracking state (covered by broader builder validation).

## Acceptance criteria

- `EntityUpdateArray` and `EntityUpdateTrackingArray` implementations MUST be available in `hipster-entity-core`.
- Bitmask-based touched tracking MUST be the canonical implementation for tracking arrays.
- Merge behavior MUST require explicit mode naming to avoid silent semantic mismatches.
- Storage variants MUST specialize for 64-field threshold to keep common case compact.
- No-op same-value assignment behavior MUST be clearly documented and tested.
