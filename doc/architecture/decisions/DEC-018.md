# DEC-018: Generator freeze marker semantics

- Status: Proposed
- Date: 2026-04-13
- Owners: project
- Related docs: [Generator freezing architecture](../gen-freezing.md), [Generator freezing brainstorm](../../brainstorm/gen-freezing.md)
- Supersedes: -
- Superseded by: -

## Context

The generator produces source code that may be manually edited or need stabilization when a bug is discovered. Without an explicit freeze mechanism, the generator can undo manual fixes, repeatedly rewrite problematic generated code, or cause churn when a developer needs to retain a local workaround.

A freeze marker allows developers to mark generated code as immutable for generator rewrites while still permitting the generator to process other parts of the same file or entity.

## Decision

1. Support a dedicated annotation marker: `@GeneratedFrozen`.
2. Support a comment marker: `// generator:freeze`.
3. Support optional region markers for future method-body or block-level freeze semantics: `//#region generator freeze` / `//#endregion`.
4. Support optional expected-fix metadata on freeze markers, such as `until="YYYY-MM-DD"`, so tooling can track when a fix should become available.
5. The generator MUST detect freeze markers on declarations and skip any source rewrite of the frozen element.
6. A file-level freeze marker SHOULD treat the whole file as immutable and avoid applying generator edits to that file.
7. The generator MUST emit diagnostics for skipped frozen elements, and tooling SHOULD emit a warning if the expected fix date has passed.

## Consequences

- Positive:
  - Developers can preserve manual fixes around buggy generator output.
  - The generator can continue updating unfrozen code in the same source tree.
  - Partial migration is safer: problematic generated artifacts can be frozen while the rest remains managed.
- Negative:
  - Frozen code can become stale or diverge from current metadata semantics.
  - Overuse of freezing may hide generator bugs and reduce coverage of future improvements.
- Follow-up:
  - Implement parser support for annotation, comment, and region markers.
  - Decide whether generated marker annotation source is emitted by tooling.
  - Add diagnostics and tests for frozen-file and frozen-element preservation.
- Add tooling alerts for overdue expected-fix metadata.

## Out of scope

- Runtime enforcement or reflection of frozen state.
- Automatic unfreeze when generator bugs are fixed.
- Using freeze markers to alter generated metadata semantics.

## Acceptance criteria

- The generator recognizes `@GeneratedFrozen` and `// generator:freeze` markers on declarations.
- The generator recognizes optional expected-fix metadata such as `until="YYYY-MM-DD"`.
- The generator skips rewriting frozen declarations and preserves their source text.
- A file-level freeze marker causes the generator to treat the file as immutable.
- Diagnostics are emitted when a frozen element is skipped.
- Tooling emits a warning when a freeze marker is overdue relative to its expected-fix metadata.
