# DEC-016: Field-name-to-ordinal dispatch must use `forName` + ordinal indexing; per-call HashMap is forbidden

- Status: Accepted
- Date: 2026-04-03
- Owners: project
- Related docs: [DEC-015](DEC-015.md), [DEC-004](DEC-004.md), [Field lookup implementation guide](../../user/field-lookup-guide.md), [ViewMeta API](../../../hipster-entity-api/src/main/java/hr/hrg/hipster/entity/api/ViewMeta.java)

## Context

Any code that parses or maps field names to field ordinals (deserializers, mappers, projection
bridges, update appliers, etc.) needs to resolve a string field name to its integer ordinal in the
values array.

A naive implementation allocates a `HashMap<String, Integer>` per invocation, populates it by
iterating `fieldNameAt(i)` inside a for-loop, and then calls `Map.get(name)` inside the parse loop.
This pattern:

- allocates a new `HashMap` on every deserialize call;
- computes hash codes for every inserted field name during setup;
- computes another hash code and performs a bucket lookup on every incoming JSON field;
- defeats JIT inlining budgets due to `HashMap.get` virtual dispatch chains;
- is completely unnecessary because a zero-allocation alternative already exists.

The same problem occurs when user code (or generated code) builds its own `String[]` name arrays
paired with ordinal int arrays and performs linear scan or binary search at runtime. Both patterns
reconstruct at runtime what is already encoded statically in the compiled switch inside
`FieldNameMapper.forName`.

## Decision

> **Per-call HashMap (or any other dynamic name→ordinal structure) construction is forbidden.**

The single permitted pattern for field-name-to-ordinal dispatch is:

1. **Pre-build a `readers[]` (or `writers[]`) array** indexed by ordinal, **once per
   deserializer/serializer instance** (not per call). The array length is `meta.fieldCount()`.

2. **Inside the parse loop**, call `meta.forName(name)` to resolve the field. This call hits a
   generated `switch` statement (see DEC-015) — O(1), zero allocation, JIT-friendly.

3. **Capture the returned field reference once** — do not call `forName` twice for the same field.
   Use `field.ordinal()` as the direct index into the pre-built array. Calling ordinal() twice
   on the same reference is acceptable; calling `forName` twice for the same field is not.

4. **Unknown field names** (`forName` returns `null`) must be handled with `p.skipChildren()` or
   equivalent sink — never by throwing.

### Canonical parse-loop shape

```java
// ── OUTSIDE the parse loop ──────────────────────────────────────────────────
// Built once per deserializer instance (or per meta, if shared). NOT rebuilt per call.
ValueReader[] readers = new ValueReader[meta.fieldCount()];
for (int i = 0; i < meta.fieldCount(); i++) {
    readers[i] = readerFor(meta.fieldTypeAt(i));
}

// ── INSIDE the parse loop ────────────────────────────────────────────────────
F field = meta.forName(name);          // compiled switch — O(1), zero allocation
if (field != null) {
    int ordinal = field.ordinal();     // capture once; use for both read and store
    values[ordinal] = readers[ordinal].read(p);
} else {
    p.skipChildren();                  // unknown field — skip, never throw
}
```

### Canonical anti-patterns (FORBIDDEN)

```java
// ❌ FORBIDDEN — HashMap rebuilt per deserialize() call
Map<String, Integer> fieldNameToOrdinal = new HashMap<>(fieldCount * 2);
for (int i = 0; i < fieldCount; i++) {
    fieldNameToOrdinal.put(meta.fieldNameAt(i), i);   // hash + bucket write per field
}
// ...
Integer ordinal = fieldNameToOrdinal.get(name);       // hash + bucket read per JSON field

// ❌ FORBIDDEN — calling forName twice for the same field
values[meta.forName(name).ordinal()] = readers[meta.forName(name).ordinal()].read(p);

// ❌ FORBIDDEN — new TypeReference<>() {} inside parse loop
Map<String, List<Long>> m = p.readValueAs(new TypeReference<>() {});

// ❌ FORBIDDEN — any dynamic structure built inside the hot path
String[] names = new String[fieldCount];              // rebuilt every call
```

### Permitted use of `TypeReference` for complex generic types

When a field type is a parameterized generic (e.g. `Map<String, List<Long>>`), a `TypeReference`
is required for deserializing it. It must be declared **`private static final`** — created once at
class-load time, never inside a method or parse loop:

```java
// ✅ CORRECT — static final, created once
private static final TypeReference<Map<String, List<Long>>> METADATA_TYPE_REF =
        new TypeReference<>() {};

// inside parse loop — TypeReference is looked up from constant pool, not allocated
case "metadata" -> values[5] = p.currentToken() == JsonToken.VALUE_NULL
        ? null
        : p.getCodec().readValue(p, METADATA_TYPE_REF);
```

## Consequences

- **Positive**:
  - Zero per-call allocation in the field-dispatch path.
  - `forName`'s compiled switch integrates cleanly into JIT's inlining pipeline.
  - Pre-built `readers[]`/`writers[]` arrays are loaded from a single reference per call with no
    further setup.
  - Parse loops are shorter, easier to audit, and free of generics or boxing surprises.
  - Static `TypeReference` constants eliminate the most common source of phantom allocation in
    generic-field deserialization.

- **Negative**:
  - Serializer/deserializer classes must store `readers[]`/`writers[]` as fields or accept them as
    constructor arguments — not a significant burden but requires a conscious design choice.
  - Implementors unfamiliar with the pattern may reach for `HashMap` by instinct; this decision
    document and the Javadoc on `ViewMeta.forName` are the primary guardrails.

## Scope

This decision governs all code that maps field names to ordinals across the project:

- Jackson deserializers (`EntityJacksonViewDeserializer`, generated boilerplate deserializers)
- Any future format adapters (CSV, binary, proto-like)
- Reflection-free mapper utilities
- Agent-generated code

Boilerplate code that uses a `switch(name)` directly on a known, fixed set of field constants
(i.e. the string literals are hardcoded by the generator) is already compliant — the `switch`
**is** the O(1) lookup. That pattern does not need to call `forName` and is also acceptable.
