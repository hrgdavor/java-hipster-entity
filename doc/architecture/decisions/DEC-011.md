# DEC-011: Automatic builder interface generation

- Status: Proposed
- Date: 2026-03-30

## Context

DEC-008 established builder policy and naming constraints. This decision focuses on generating builder interfaces as stable contracts that remain compatible with multiple implementation strategies (proxy or generated concrete).

## Decision

Generate builder interfaces that preserve record-style naming consistency and remain compatible with projection-only materialization.

**Base pattern:**
For each view interface `PersonSummary`, generate a builder interface with mutators for each logical property:

```java
public interface PersonSummaryBuilder {
    PersonSummaryBuilder id(Long value);
    PersonSummaryBuilder firstName(String value);
    PersonSummaryBuilder lastName(String value);

    PersonSummary build();
}
```

**Required methods:**
- One mutator per logical property
- `build()` terminal method
- Source-ingestion guarantee in implementation via either static factory `PersonSummaryBuilders.from(source)` or implementation constructor

**Optional methods:**
- `mergeFrom(ViewType source)`
- `clear(Property)` or `clearAll()`
- Patch helpers such as non-null merge variants

**Naming and determinism:**
- Mutator names match logical property names exactly
- Deterministic method ordering across generation runs
- Avoid JavaBean setter prefixes unless explicitly configured
- No renaming based on storage backend details

## Consequences

- Preserves consistency with record-aligned naming and DEC-008 policy.
- Enables multiple implementation strategies (proxy-backed or generated concrete) with guaranteed API compatibility.
- Allows per-view override of implementation strategy for hot path optimization.
- Requires testing rigor: all implementations must pass shared TCK-style contract suite.
- Composition complexity if builder interfaces are used in multiple contexts (may require adapter types).

## Out of scope

- Storage model for update tracking internals — covered by DEC-012.
- Optional runtime factory/selection policy details — covered by DEC-013.
- Implementation strategies or performance tuning — those are follow-up decisions.

## Acceptance criteria

- Generated builder interface MUST have one mutator per logical property and a `build()` terminal method.
- Mutator names MUST match logical property names exactly with deterministic ordering.
- Source-ingestion guarantee MUST be honored in all implementation classes via factory or constructor.
- All implementation strategies (proxy and generated concrete) MUST pass shared contract test suite.
