# DEC-013: Optional per-view implementation selection factory

- Status: Proposed
- Date: 2026-03-30
- Owners: project
- Related docs: [DEC-013 brainstorm](../../brainstorm/dec-013-factory-and-implementation-selection.md), [DEC-010 (proxy view)](DEC-010.md), [DEC-011 (builder interface)](DEC-011.md)

## Context

The platform is intentionally hybrid: proxy implementations are fast to ship and flexible, while generated concrete implementations can be faster in hot paths. Selection should be configurable per view, not only as a global runtime mode. The factory layer itself should remain optional and separable from the metadata core.

## Decision

Provide an optional runtime factory module that allows proxy variants to be replaced by generated implementations on a per-view basis:

**Core principle:**
This strategy is explicitly NOT a hard requirement of the `hipster-entity` model. The core value is interface-first metadata that enables tooling. Projects may consume metadata directly and skip the factory module entirely.

**Goals:**
- One stable API for adopters that choose to use the optional runtime factory module
- Explicit and predictable selection policy
- Safe fallback behavior (proxy default if generated is unavailable)
- Measurable criteria for when generated implementations should be used
- Per-view override support with deterministic precedence
- Ability for projects to bypass the factory layer and use metadata directly

**Candidate architecture:**

Option A: Central service factory
- Single entrypoint `EntityRuntimeFactory` with `createView`, `createBuilder`, `createBuilderFrom` methods
- Easy policy centralization; can be packaged as separate optional module
- Risk: broad interface can grow too large over time

Option B: Specialized factories
- Separate `ViewFactory`, `BuilderFactory`, `MapperFactory`
- Cleaner boundaries and simpler per-factory contracts
- More composition burden

Option C: Registry with pluggable providers
- Policy resolved through provider chain: generated provider first, proxy fallback
- Extensible by module and test-friendly
- Risk: provider ordering bugs can be subtle

**Selection policy candidates:**
1. Static configuration (module-level flag chooses proxy or generated)
2. Per-view override (view-level rule overrides module default for hot paths)
3. Capability-based (generated if present, else proxy fallback per view)
4. Performance-profile guided (proxy in dev/test, generated in production)

## Consequences

- Enables performance optimization for critical views without affecting whole-system architecture.
- Preserves flexibility to migrate from proxy to generated on a selective basis.
- Introduces optional module that applications can ignore entirely.
- Increases generator responsibility for multi-mode compatibility testing.
- Requires clear fallback semantics to avoid runtime surprises.
- Per-view overriding adds configuration complexity; must be carefully designed for developer ergonomics.

## Out of scope

- Interface contract for views or builders themselves (covered by DEC-001, DEC-008, DEC-011).
- Implementation strategies for proxy or generated code (covered by DEC-010, DEC-011).
- Build-time generation or compilation (covered by DEC-004, DEC-009).

## Acceptance criteria

- Factory module MUST be optional and separable from metadata core.
- Projects MUST be able to bypass factory entirely and use generated metadata directly.
- One stable public API MUST exist (choice of architecture pending).
- Fallback behavior MUST be explicit and safe (proxy fallback if generated unavailable).
- Per-view override support MUST have deterministic precedence rules.
- Selection policy and measurable criteria MUST be documented before full acceptance.
