# DEC-017: Identifiable<ID> as opt-in identity mixin; id() removed from ViewReader

- Status: Accepted
- Date: 2026-04-03
- Owners: project
- Related docs: [DEC-001 (interface-first model)](DEC-001.md), [DEC-010 (proxy view bridge)](DEC-010.md), [Entity API README](../../../hipster-entity-api/README.md)
- Supersedes: -

## Context

Originally `id()` was declared on `ViewReader<ID, T, F>`, forcing every readable view to advertise an identity contract. As the project evolved toward supporting embedded sub-documents (address, line items, etc.) as first-class `ViewReader`-typed views, this became incorrect: an embedded fragment has no primary key, so `id()` on its reader interface is misleading and potentially dangerous (any code accepting `ViewReader` would believe it can retrieve an id, but the value may be a data field like `street`).

A transitional experiment of placing `id()` on `EntityBase<ID>` instead of `ViewReader` was tried and rejected for the same reason: it forced every `EntityBase` implementor — including trivial marker types and test stubs — to implement `id()`.

## Decision

1. **Remove `id()` from `ViewReader`** — field-readability and identity are orthogonal concepts.
2. **Create `Identifiable<ID>`** (in `hipster-entity-api`) — a single-method mixin interface: `ID id()`.
3. **Root entity interfaces extend both `EntityBase<ID>` and `Identifiable<ID>`**:
   ```java
   public interface PersonEntity extends EntityBase<Long>, Identifiable<Long> {}
   ```
4. **Embedded/sub-document views extend `EntityBase<Void>`** (or another typed bound) without `Identifiable`:
   ```java
   public interface AddressView extends EntityBase<Void> {
       String street();
       String city();
   }
   ```
5. **Concrete array classes (`EntityReadArray`, `EntityUpdateArray`, `EntityUpdateTrackingArray`) do NOT implement `Identifiable`** — they are neutral backing stores, not domain contracts. `id()` was removed from them entirely.
6. **Proxy dispatch (`ArrayBackedViewProxyFactory`) no longer special-cases `id()`** — `id` is routed through `FieldNameMapper` like any other accessor. This only works if the field enum has an `id` constant, which is the case for all root entity enums.

## Consequences

- Root entity views cleanly advertise `id()` via `Identifiable`.
- Embedded sub-document views carry zero identity contract, preventing accidental misuse.
- `EEnumSetTrackingJmhBenchmark` inner stubs no longer need any method body — they implement `EntityBase<Integer>` with no methods.
- `FieldNameMapper` and field enum constants remain the sole resolver for all accessor dispatch including `id`.
- Any future code that generically accepts `Identifiable<ID>` will only receive types that explicitly declare identity intent.

## Acceptance criteria

- `Identifiable<ID>` MUST exist in `hipster-entity-api` as a standalone interface with only `ID id()`.
- `ViewReader` MUST NOT declare `id()`.
- Root entity marker interfaces MUST extend `Identifiable<ID>`.
- Embedded view interfaces MUST use `EntityBase<Void>` (or a non-`Identifiable` bound) to signal no identity.
- `ArrayBackedViewProxyFactory` proxy dispatch MUST NOT contain a hardcoded `"id"` name check.
