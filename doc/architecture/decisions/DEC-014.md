## Context

Update tracking and change detection are frequent operations in proxy and builder workflows. Baseline solutions (interface-typed tracking arrays or generic collections like `HashSet<Integer>`) add virtual dispatch or hashing overhead on small, frequent operations. Without specialization, tracking can become measurable in high-volume scenarios.

The project already uses bitmask-based tracking semantics from DEC-012. The remaining question is how to minimize dispatch cost on hot paths like `mark()` and `unmark()` without sacrificing code maintainability or portability.

## Decision

Use custom `EEnumSet*` and `EEnumSetBuilder*` classes with concrete dispatch specialization for update tracking arrays in proxies:

- **Immutable variant** (`EEnumSet64`, `EEnumSetLarge`): snapshot representation of tracked fields
- **Mutable variant** (`EEnumSetBuilder64`, `EEnumSetBuilderLarge`): accumulator for mark/unmark/clear operations
- **Tracking arrays** (`EntityUpdateTrackingArray64`, `EntityUpdateTrackingArrayLarge`): hold concrete builder references for static binding

Two storage variants are provided:
- 64-value enums: single `long` bitmask (no allocation, cache-friendly)
- larger enums: segmented `long[]` (predictable scaling)

Tracking arrays store concrete types rather than interface-typed references, enabling generated code to use static binding and reduce virtual method call overhead on hot paths.

This strategy is **optional** and scoped to update tracking performance optimization. Projects may use interface-based tracking or alternative approaches for change detection.

## Consequences

- **Positive**:
  - Reduces dispatch cost on high-frequency small operations (bit flipping) by enabling static binding in C2 inlining scenarios
  - Predictable memory layout without auxiliary collection overhead
  - Embeddable design allows code generation to inline whole tracking structures into proxy fields
  - Measurable via JMH benchmarks; dispatch benefit is validated empirically rather than assumed
  - Directional measurements show 3–9% improvement on mutation paths (mark/unmark) with concrete dispatch

- **Negative**:
  - Increases code generation complexity (choice and instantiation of correct variant)
  - Proliferates types (separate classes for each role and width)
  - Tight coupling between tracking arrays and concrete enum-set types
  - Allocation cost when transitioning from 64-bit to segmented storage
  - No dynamic fallback; strategy is fixed per instance

## Out of scope

- Snapshot creation cost — allocation and conversion are separate optimization concerns; dispatch does not dominate snapshot operations.
- One-size-fits-all tracking for all use cases — this decision addresses hot paths in proxy and builder workflows; other scenarios may use different strategies.
- Generic collection performance — this decision compares against interface dispatch and allocation overhead, not alternative collection frameworks.

## Acceptance criteria

- Custom `EEnumSet*` and `EEnumSetBuilder*` classes MUST be implemented in `hipster-entity-core` with storage variants for 64 and larger enums.
- Tracking arrays MUST hold concrete builder references to enable static binding.
- JMH benchmarks (`EEnumSetTrackingJmhBenchmark`) MUST validate that concrete dispatch provides measurable improvement on mark/unmark operations.
- Implementation guide MUST document the choice tradeoffs, when to use this strategy, and when to reassess.
